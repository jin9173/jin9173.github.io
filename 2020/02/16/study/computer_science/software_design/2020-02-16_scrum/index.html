소프트웨어 생명주기

  • 소프트웨어 생명주기: 개발단계, 단계별 주요활동, 활동의 결과를 산출물로 표현

  • 소프트웨어 생명주기 모형

    • 폭포수 모형(고전적): 단계가 끝난 후 다음 단계 진행 (각 단계들은 병행하여 수행X)

      [타당성 검토 → 계획→ 요구분석→ 설계→ 구현(코딩)→ 시험(검사)→ 유지보수]

    • 프로토타입 모형: 견본품을 바탕으로 최종 결과물 예측 (폭포수 모형의 요구분석 단계 보완)

      [요구수집→ 빠른 설계→ 프로토타입 구축→ 고객 평가→ 프로토타입 조정→ 구현]

    • 나선형 모형(점진적): 위험 분석 기능 (폭포수모형과 프로토타입 모형 보완) [유지보수 과정X]

    • 애자일 모형: 고객의 요구사항을 수용하며 스프린트(짧은 개발 주기) 반복 [일정 주기가 끝날때마다 테스트]

스크럼(Scrum) 기법

: 스크럼이란 팀이 중심이되어 개발의 효율성을 높인다는 의미

  • 팀원 스스로가 스크럼 팀을 구성
  • 팀원 스스로 해결

스크럼 팀

1. 제품 책임자(PO; Project Owner)

  • 의뢰자나 사용자가 요구사항을 책임지고 의사 결정
  • 요구사항이 담긴 백로그(Backlog)를 작성하고 우선순위 지정 (팀원들 개입 불가능)
  • 제품에 대한 테스트를 수행하면서 주기적으로 요구사항의 우선순위 갱신

2. 스크럼 마스터(SM; Scrum Master)

  • 객관적인 시각에서 조언을 해주는 가이드 역할 수행 (팀원들 통제X)
  • 일일 스크럼 회의를 통해 진행 사항 점검
  • 개발 과정에서 발생된 장애요소를 공론화하여 처리

3. 개발팀(DT; Development Team)

: 제품 책임자와 스크럼 마스터를 제외한 모든 팀원 (7~8명)


스크럼 개발 프로세스

1. 제품 백로그(Product Backlog)

: 제품 개발에 필요한 모든 요구사항(User Story)을 우선순위에 따라 나열한 목록

  • 지속적으로 업데이트
  • 사용자 스토리를 기반으로 전체 일정 계획인 릴리즈 계획을 수립

2. 스프린트 계획 회의(Sprint Planning Meeting)

: 제품 백로그 중 이번 스프린트에서 수행할 작업을 대상으로 단기 일정을 수립

  • 요구사항(User Story)을 테스크 작업단위로 분할한 후 개발자별로 수행할 작업 목록인 스프린트 백로그를 작성

3. 스프린트(Sprint)

: 2~4주 기간 내의 실제 개발작업 진행 과정

  • 테스크 당 작업 시간 추정 후 할당 (개발자가 원하는 테스크 직접 선별)
  • 할일(To Do), 진행중(In Progress), 완료(Done) 상태로 테스크 분류

4. 일일 스크럼 회의(Daily Scrum Meeting)

: 매일 15분동안 서서 진행 상황 점검

  • 남은 작업시간은 소멸차트(Burn-down Chart)에 표시
  • 발견된 장애요소는 스크럼 마스터가 해결할 수 있도록 도와줌

5. 스프린트 검토 회의(Sprint Review)

: 사용자가 포함된 참석자 앞에서 한시간 내에 테스팅 수행

  • 제품 책임자는 개선할 사항에 대한 피드백을 정리한 후 다음 스프린트에 반영할 수 있도록 제품 백로그를 업데이트

6. 스프린트 회고(Sprint Retrospective)

: 해당 스프린트가 끝난 시점에서 수행하거나 일정 주기로 정해놓은 규칙을 잘 준수했는지, 개선할 점은 없는지 등을 확인하고 기록
