디자인 패턴

: 어떤 프로그램을 구현할 때 효율적인 구현 방식에 대한 규격을 패턴화시킨 것


MVC 디자인 패턴

: 모델(Model)-뷰(View)-컨트롤러(Controller) 디자인 패턴은 사용자 인터페이스와 비즈니스 로직을 분리하여 개발하는 방법론이다. 즉, 이는 Frontend(인터페이스)와 Backend(데이터베이스)를 나눈 시발점이라고 할 수 있다.

즉, HTML코드와 데이터베이스를 만들어 그 데이터를 HTML안에 넣는 코드를 분리시켜 유지보수를 쉽게 만들어 생산성(비즈니스 로직)을 높였다. (어떤 코드들은 데이터베이스를 통신하는 역할만 수행하고, 어떤 코드들은 데이터베이스를 구성하는 역할만 수행하고, 어떤 코드는 사용자를 위한 UI를 만들어주는 역할만 수행한다.)

  • Model: DB 테이블 정의
  • View: 사용자가 보게 될 화면의 모습 정의
  • Controller: 애플리케이션의 제어 흐름 및 로직을 정의
    • DB의 데이터를 사용자가 화면으로 전달
    • 사용자의 데이터를 적절히 가공해서 DB에 변경사항을 전달
    • DB에 한정되지 않고 그 밖에도 많은 역할을 한다

이렇게 3가지로 나눠서 개발을 진행하면 모델, 뷰, 템플릿 모듈 간에 독립성을 유지할 수 있고, 소프트웨어 개발의 중요한 원칙인 느슨한 결합 설계의 원칙에도 부합한다. 또한 화면 디자이너, 응용 개발자, DB 설계자 간 협업도 쉬워진다.

MTV 디자인 패턴

: 장고에서 사용하는 디자인 패턴

장고에서 프로젝트를 생성하기 위해 startproject 및 startapp 명령을 실행하면, 자동으로 프로젝트 뼈대에 해당하는 디렉터리와 파일들을 만들어준다.

  • Model (models.py)
  • Template: MVC의 View와 같음 (templates>.html)
  • View: MVC의 Controller와 같음 (views.py)

MTV 코딩 순서

  1. 프로젝트 뼈대 만들기: 프로젝트 및 앱 개발에 필요한 디렉터리와 파일 생성(startproject, startapp)
  2. 모델 코딩하기: 테이블 관련 사항을 개발(models.py, admin.py)
  3. URLconf 코딩하기: URL 및 뷰 매핑 관계를 정의(urls.py)
  4. 뷰 코딩하기: 애플리케이션 로직 개발(views.py)
  5. 템플릿 코딩하기: 화면 UI 개발 (templates>.html)

settings.py 주요 사항

: settings.py 는 프로젝트 설정 파일이다. 처음 프로젝트를 생성하면 장고가 기본 사항들을 자동으로 등록해주므로, 추가적으로 필요한 사항은 개발을 진행하면서 개발자가 원하는 내용을 등록하면 된다.

필수 사항

  1. 데이터베이스 설정: default로 SQLite3 데이터베이스 엔진을 사용하는 것으로 지정

    만일 MySQL이나 Oracle 등 다른 데이터베이스로 변경하고 싶다면 settings.py 파일에서 수정해주면 됨

  2. 애플리케이션 등록: 개발하는 앱(프로젝트)에 포함되는 애플리케이션들을 모두 설정 파일에 등록

  3. 템플릿 항목 설정: TEMPLATES 항목으로 지정 (장고가 찾을 수 있도록 연결)

  4. 정적 파일 항목 설정: STATIC_URL 등 관련 항목을 지정

  5. 타임존 지정: default값으로 세계표준시(UTC)로 설정되어 있는데 한국시간으로 변경하길 권장


models.py 주요 사항

: models.py 는 테이블을 정의하는 파일이다. 장고의 특징 중 하나로, 데이터베이스 처리는 ORM(Object Relational Mapping)기법을 사용한다.

즉, 테이블은 클래스로 매핑해서 테이블에 대한 CRUD(Create, Read, Update, Delete) 기능을 클래스 객체에 대해 수행하면, 장고가 내부적으로 SQL 처리를 하여 데이터베이스를 반영해주는 방식이다.

장고에서는 마이그레이션 개념이 있는데 이는 테이블 및 필드의 생성, 삭제, 변경 등과 같은 데이터베이스에 대한 변경 사항을 알려주는 정보이다.


URLconf 주요 사항

: URL과 뷰를 매핑해주는 urls.py 파일을 말한다. (URLconf를 정의할 때는 하나의 파일에 정의할 수도 있고, 2개의 파일에 정의할 수도 있다.)

2개의 파일에 URLconf를 정의하면 프로젝트 URL과 앱 URL로 나누어 2계층으로 나눠서 코딩하기 때문에 변경이 쉽고 확장도 용이하다.


views.py 주요 사항

: views.py 는 뷰로직을 코딩하는 가장 중요한 파일이다. (가독성, 유지보수 편리성, 재활용성 필수로 고려)

  • 함수형 뷰

  • 클래스형 뷰

    : 클래스형 뷰를 사용하면 장고가 제공하는 제네릭 뷰를 사용할 수 있고, 재활용 및 확장성 측면에서 유리


templates 주요 사항

: 웹 화면(페이지)별로 템플릿 파일들이 하나씩 필요하므로 템플릿 디렉터리가 필요하다.

  • 프로젝트 템플릿 디렉터리

    : TEMPLATES 설정의 DIRS 항목에 지정된 디렉터리 (base.html)

  • 앱 템플릿 디렉터리

    : 각 애플리케이션 디렉터리마다 존재하는 templates/ 디렉터리


Admin 사이트

: 테이블안의 콘텐츠를 열람하고 수정하는 기능을 제공 (admin.py)


개발용 웹 서버 runserver

: 개발 과정에서 작성된 코드를 실행하고 테스트할 수 있다.

실제 고객에게 상용화를 고려한다면, 처리 능력도 떨어지고 보안에 취약한 runserver 대신 Apache 또는 Nginx 등의 상용 웹 서버를 사용하기를 권장한다.
